## 1. GitHub Actions 사용 검토

### 현황

| 워크플로우 | 실행 주기 | 연간 예상 실행 횟수 |
|-----------|----------|-------------------|
| `collect-realtime.yml` | 15분마다 | ~35,040회 |
| `collect-environment.yml` | 매시간 | ~8,760회 |
| `collect-static.yml` | 1일 1회 | ~365회 |
| `cleanup.yml` | 1일 1회 | ~365회 |
| `deploy.yml` | 이벤트 기반 | 가변적 |

### GitHub 약관 관련 조항

GitHub Terms for Additional Products에 따르면 Actions는 다음 용도로 사용할 수 없습니다:

> - 크립토마이닝
> - 서비스, 장치, 데이터 등에 대한 무단 접근 시도
> - 서버에 "불균형한(disproportionate)" 부담을 주는 활동
> - GitHub 호스팅 러너 사용 시, **소프트웨어 프로젝트의 프로덕션, 테스트, 배포 또는 퍼블리싱과 무관한 활동**

### 판정: ⚠️ 주의 필요

- **긍정적 요소**:
  - 모든 워크플로우에 `timeout-minutes: 3` 설정 (리소스 낭비 방지)
  - npm 캐시 활용으로 의존성 설치 최적화
  - 소프트웨어 프로젝트의 데이터 수집 및 배포 목적 (약관 허용 범위)
  - 크립토마이닝, CDN, 서버리스 등 금지 용도 아님

- **주의 사항**:
  - 15분 간격 실행은 연간 ~35,040회로 상당히 높은 빈도
  - 무료 플랜 기준 GitHub Actions 월 2,000분 한도 존재 — 현재 빈도라면 월 ~2,880회 × 약 1~2분 = 약 2,880~5,760분으로 **무료 한도를 초과할 가능성**
  - 매 수집마다 PR 생성 → 자동 병합 패턴은 불필요한 오버헤드 발생

- **개선 권장**:
  - PR 기반 자동 병합 대신 직접 커밋 방식 고려 (PR 노이즈 감소)
  - 무료 플랜 사용 시 실행 빈도 검토 (유료 플랜이라면 문제 없음)

## 2. GitHub Pages 사용 검토

### 현황
- Next.js 14 정적 사이트 (`output: 'export'`)
- 커스텀 도메인 `nadeuri.today` 사용
- 공개 접근 가능한 공익 정보 제공

### GitHub 약관 관련 조항

> GitHub Pages는 "개인 및 조직 프로젝트를 보여주기 위한 정적 웹 페이지 호스팅"을 목적으로 합니다.
>
> 다음 용도로는 사용할 수 없습니다:
> - 온라인 비즈니스 운영
> - 전자상거래 사이트
> - 상업적 거래를 촉진하거나 상업적 SaaS를 제공하는 웹사이트

### 판정: ✅ 준수

- 비상업적 공익 프로젝트 (서울 지하철 접근성 시설 정보)
- 정적 사이트 배포 — Pages의 의도된 사용 목적에 부합
- 상업적 거래나 SaaS 제공 없음
- 대역폭: 정적 JSON + HTML이므로 과도한 트래픽 가능성 낮음

## 3. 저장소 데이터 저장 검토

### 현황
- 수집 데이터를 `data/` 디렉토리에 JSON 파일로 저장
- 7일 보관 후 자동 삭제 (`cleanup.yml`)
- 현재 데이터 용량: 약 2.2MB
- 7일 최대 누적: 약 27MB

### GitHub 약관 관련 조항

> "인프라에 부당한 부담을 주는 것으로 판단되는 저장소"는 삭제될 수 있습니다.

GitHub 권장사항:
- 저장소 크기: 이상적으로 1GB 미만
- 개별 파일: 100MB 미만

### 판정: ✅ 준수

- 27MB 최대 용량은 GitHub 권장 한도(1GB) 대비 미미
- 자동 정리 정책으로 무제한 증가 방지
- Git 히스토리에 삭제된 데이터가 남지만, 정기적 관리 시 허용 범위

## 7. 외부 API 사용 및 데이터 출처 검토

### 현황
- **서울 열린데이터광장 API** (openapi.seoul.go.kr) 사용
- 공식 API 키 인증 방식
- 하루 약 121회 API 호출 (실시간 96회 + 환경 24회 + 정적 1회)

### 판정: ⚠️ 확인 필요

- **긍정적 요소**:
  - 웹 스크래핑 아닌 공식 API 사용
  - 재시도 시 지수 백오프 적용 (1초, 2초, 4초)
  - 요청 타임아웃 10초 설정
  - API 출처 README에 명시

- **확인 필요 사항**:
  - 서울시 열린데이터 API 이용약관에서 15분 간격 호출이 허용되는지 확인
  - API 일일 호출 한도 확인 (서울시 API는 일반적으로 일 1,000회 제한 있음)
  - 수집 데이터의 재배포 조건 확인 (공공데이터법에 따른 자유이용 가능 여부)
  - HTTP(비암호화) 통신 사용 — 보안 관점에서 HTTPS 전환 권장

## 8. 기타 검토 항목

### Git 히스토리 관리
- 15분마다 자동 커밋이 발생하여 Git 히스토리가 빠르게 증가
- `.git` 디렉토리 크기가 시간이 지남에 따라 상당히 커질 수 있음
- `git gc` 또는 주기적인 히스토리 정리 필요할 수 있음

### GitHub API 간접 사용
- `gh pr create`, `gh pr merge` 등 GitHub CLI 사용
- 15분마다 PR 생성/병합은 GitHub API 레이트 리밋에 영향 가능
- 직접 커밋 방식으로 전환하면 API 호출 감소 가능

## 종합 결론

### 위반 여부: **약관 위반 없음**

이 레포지토리는 GitHub Terms of Service를 준수하고 있습니다. 발견된 사항은 "위반"이 아닌 "최적화 권장" 수준입니다.

### 개선 권장 사항 (우선순위순)

1. **[중요]** 서울시 열린데이터 API 이용약관 확인 — 호출 빈도 및 데이터 재배포 조건
2. **[권장]** GitHub Actions 무료 플랜 분(minutes) 한도 모니터링 — 15분 간격 실행이 한도를 초과할 수 있음
3. **[권장]** PR 기반 자동 병합 → 직접 커밋 방식 전환 검토 — GitHub API 호출 및 PR 노이즈 감소
4. **[선택]** 서울시 API 호출 시 HTTP → HTTPS 전환 검토
5. **[선택]** Git 히스토리 크기 모니터링 및 정기적 정리 계획 수립
